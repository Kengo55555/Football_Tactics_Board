## 1. 全体像 — アプリが動くまでの流れ

まず、ユーザーがブラウザで画面を見るまでに何が起きているかを大まかにつかみましょう。

```
開発者がコードを書く
    ↓
npm run dev（Vite が起動）
    ↓
Vite が JSX → JavaScript に変換、Tailwind → CSS に変換
    ↓
ブラウザが変換済みのファイルを受け取る
    ↓
React がページ内の <div id="root"> に画面を組み立てる
    ↓
ユーザーが操作するたびに React が必要な箇所だけ再描画
```

このアプリでは以下のファイルが順番に読み込まれます。

```
index.html → src/main.jsx → src/App.jsx → src/tactics-board_1.jsx
```

`index.html` を見てみましょう。

```html
<!-- index.html -->
<body>
  <div id="root"></div>                              ← ここは空っぽ
  <script type="module" src="/src/main.jsx"></script> ← ここから React が起動
</body>
```

`<div id="root">` の中身は空です。React が JavaScript で中身を生成して埋め込みます。これが「シングルページアプリケーション（SPA）」と呼ばれる仕組みです。

## 2. JavaScript — ブラウザを動かす言語

### 何ができるのか

JavaScript はブラウザの中で動くプログラミング言語です。HTML が「文書の構造」、CSS が「見た目」を担当するのに対し、JavaScript は「動き」を担当します。

```
HTML  → 「ここにボタンがある」「ここに画像がある」 → 骨組み
CSS   → 「ボタンは青色で角丸」                     → 見た目
JavaScript → 「ボタンを押したら選手が動く」         → 動作
```

### 本アプリでの使われ方

本アプリの動作はすべて JavaScript で書かれています。例えば：

- 選手をドラッグで動かす処理
- 矢印を描く処理
- 保存・読み込みの処理
- シナリオ（試合全体 / 守備 / 攻撃）の切り替え

### 基本の書き方（変数と関数）

```javascript
// 変数: データに名前を付けて保存する箱
const pitchWidth = 50;     // 変わらない値は const
let playerName = 'GK';    // 後で変わる値は let

// 関数: 処理をまとめて名前を付けたもの
function greet(name) {
  return `こんにちは、${name}さん`;
}

// アロー関数: 関数の短い書き方（本アプリで多用）
const greet = (name) => `こんにちは、${name}さん`;
```

本アプリの `src/tactics-board_1.jsx` で実際に使われている例：

```javascript
// ピッチサイズを定数として定義（55行目付近）
const PITCH_W = 50;
const PITCH_H = 68;

// 写真をリサイズする関数（9行目付近）
const resizePhoto = (file, size = 220) => new Promise((resolve, reject) => {
  // ... 処理 ...
});
```

### よく出てくる記法

| 記法 | 意味 | 例 |
|------|------|------|
| `const` | 再代入できない変数宣言 | `const x = 10;` |
| `let` | 再代入できる変数宣言 | `let y = 20; y = 30;` |
| `=>` | アロー関数（関数の短縮記法） | `(a, b) => a + b` |
| `` `${}` `` | テンプレートリテラル（文字列に変数を埋め込む） | `` `名前は${name}` `` |
| `...` | スプレッド構文（配列やオブジェクトを展開） | `{...player, x: 10}` |
| `?.` | オプショナルチェーン（null なら止まる） | `obj?.name` |

## 3. JSX — HTML と JavaScript の合体

### JSX とは

JSX（JavaScript XML）は、JavaScript の中に HTML のようなタグを直接書ける構文です。React 専用の書き方で、ブラウザはそのまま読めません。Vite が通常の JavaScript に変換します。

### 見た目は HTML、中身は JavaScript

```jsx
// JSX で書いたコード（開発者が書く）
<button onClick={() => setZoom('full')}>試合全体</button>

// Vite が変換した後（ブラウザが受け取る）
React.createElement('button', { onClick: () => setZoom('full') }, '試合全体');
```

どちらも同じ意味ですが、JSX の方が圧倒的に読みやすいです。

### HTML との違い

JSX は HTML に似ていますが、いくつかの違いがあります。

| HTML | JSX | 理由 |
|------|-----|------|
| `class="btn"` | `className="btn"` | `class` は JavaScript の予約語だから |
| `for="name"` | `htmlFor="name"` | `for` は JavaScript の予約語だから |
| `style="color: red"` | `style={{ color: 'red' }}` | JavaScript のオブジェクトで渡すから |
| `onclick="..."` | `onClick={...}` | キャメルケース（先頭小文字 + 途中大文字）にするルール |

### 本アプリでの実例

`src/tactics-board_1.jsx` から、選手を描画している部分を見てみましょう。

```jsx
{players.map(p => (
  <div
    className="relative rounded-full flex items-center justify-center"
    style={{
      width: zoom === 'full' ? 32 : 56,    // 全体表示なら小さく、シーン表示なら大きく
      height: zoom === 'full' ? 32 : 56,
      background: isHome ? HOME_COLOR : AWAY_COLOR,
    }}
  >
    {p.photo ? (
      <img src={p.photo} alt={p.name} />    // 写真があれば表示
    ) : (
      <span>{p.num}</span>                   // なければ背番号を表示
    )}
  </div>
))}
```

ここでのポイント：

- `{players.map(p => ...)}` — JavaScript の配列メソッドで選手を繰り返し描画
- `{p.photo ? ... : ...}` — 条件によって表示内容を切り替え（三項演算子）
- `style={{ width: zoom === 'full' ? 32 : 56 }}` — 状態に応じてサイズを動的に変更

**HTML では不可能な「データに応じて自動で画面を変える」ことが、JSX なら自然に書けます。**

## 4. React — 画面を自動で更新する仕組み

### React が解決する問題

ブラウザの画面を JavaScript だけで更新しようとすると、こうなります。

```javascript
// React なし: 手動で DOM を操作
document.getElementById('player1').style.left = '100px';
document.getElementById('player1').style.top = '200px';
document.getElementById('player1-name').textContent = '太郎';
// ... 16人分 × 複数のプロパティ → 管理が大変
```

React があると：

```jsx
// React あり: 状態を変えれば画面は勝手に更新される
setPlayers(prev => prev.map(p =>
  p.id === 'h1' ? { ...p, x: 100, y: 200, name: '太郎' } : p
));
// → React が差分を検出して、変わった部分だけ画面を更新
```

### コンポーネントという考え方

React では画面を「コンポーネント」という部品に分けて組み立てます。レゴブロックのようなイメージです。

```
TacticsBoard（アプリ全体）
├── Header（ヘッダー：保存・履歴・リセットボタン）
├── Toolbar（ツールバー：シナリオ切替・ツール選択）
│   ├── ScenarioBtn（「試合全体」などのボタン）
│   └── ToolBtn（「動かす」「矢印」ボタン）
├── Field（ピッチ：選手・ボール・矢印）
│   ├── Player × 16（各選手）
│   ├── SoccerBall（ボール）
│   └── Arrow（矢印）
├── SidePanel（右パネル：選手リスト）
│   └── TeamTab（チームタブ）
└── Modal（保存ダイアログ・履歴モーダル）
    └── SaveCard（保存データ1件分のカード）
```

各コンポーネントは独立した部品なので、`SoccerBall` を修正しても他の部品には影響しません。

### コンポーネントの書き方

本アプリの `SoccerBall` コンポーネントを見てみましょう。

```jsx
function SoccerBall({ size = 28 }) {
  return (
    <svg width={size} height={size} viewBox="0 0 36 36">
      {/* ボールの SVG 描画 */}
    </svg>
  );
}
```

- `function SoccerBall` — 関数として定義する（関数コンポーネント）
- `{ size = 28 }` — 外から渡されるデータ（**props** と呼ぶ）。`size` が指定されなければ 28 を使う
- `return (...)` — この部品の見た目を JSX で返す

使う側はこう書きます。

```jsx
<SoccerBall size={zoom === 'full' ? 20 : 36} />
```

HTML のタグのように自作の部品を配置できる、これが React の基本です。

## 5. Hooks — React の便利な道具箱

### Hooks とは

Hooks は React が用意している「状態管理」や「副作用処理」のための関数群です。名前が必ず `use` で始まります。

本アプリで使われている Hooks：

| Hook | 役割 | 使用例 |
|------|------|--------|
| `useState` | データを記憶し、変わったら画面を更新する | 選手の位置、ズーム状態 |
| `useRef` | 再描画せずに値を保持する / DOM 要素を参照する | ピッチの DOM 要素 |
| `useCallback` | 関数をメモ化（毎回作り直さない）する | イベントハンドラ |
| `useEffect` | 画面描画後に何かを実行する | 保存データの読み込み |

### useState — 最も重要な Hook

`useState` は「覚えておきたい値」と「その値を更新する関数」のペアを返します。

```jsx
// src/tactics-board_1.jsx 79行目付近
const [zoom, setZoom] = useState('full');
//      ↑        ↑                  ↑
//    現在の値  更新する関数     初期値
```

`setZoom('home')` を呼ぶと：
1. `zoom` の値が `'full'` → `'home'` に変わる
2. React が「zoom を使っている部分」を自動で見つけて再描画する
3. ピッチが自陣ハーフの表示に切り替わる

**開発者は「値を変える」だけで、「画面のどこを更新するか」を考える必要がありません。**

### useRef — 再描画しない記憶

`useState` で値を変えると画面が再描画されますが、再描画不要な値には `useRef` を使います。

```jsx
const fieldRef = useRef(null);   // ピッチの DOM 要素を参照するため
const pendingPhotoIdRef = useRef(null);  // 写真アップロード対象の選手 ID
```

`fieldRef.current` にピッチの実際の HTML 要素が入り、ドラッグ座標の計算などに使います。

### useEffect — 「画面が出た後にやること」

```jsx
useEffect(() => {
  // 画面が描画された後に実行される
  // 例: 保存データを localStorage から読み込む
}, []);  // [] は「初回の1回だけ実行」の意味
```

`[]`（依存配列）の中身によって実行タイミングが変わります。

| 依存配列 | いつ実行されるか |
|----------|-----------------|
| `[]` | 初回の描画後に1回だけ |
| `[zoom]` | `zoom` が変わるたびに |
| 省略 | 毎回の描画後に |

## 6. Tailwind CSS — クラス名で見た目を決める

### CSS の基本

CSS（Cascading Style Sheets）は HTML の見た目を指定する言語です。

```css
/* 通常の CSS */
.button {
  padding: 8px 16px;      /* 内側の余白 */
  border-radius: 12px;    /* 角丸 */
  font-weight: bold;      /* 太字 */
  background: #1e3a8a;    /* 背景色 */
}
```

### Tailwind の考え方

Tailwind CSS は、あらかじめ大量の「1つだけ仕事をするクラス」を用意しています。CSS ファイルを書く代わりに、クラス名を組み合わせてスタイルを作ります。

```jsx
// Tailwind を使う場合（CSS ファイル不要）
<button className="px-4 py-2 rounded-xl font-bold bg-blue-800">
  保存
</button>
```

| クラス名 | 意味 | 通常の CSS で書くと |
|----------|------|---------------------|
| `px-4` | 左右の padding を 16px に | `padding-left: 16px; padding-right: 16px;` |
| `py-2` | 上下の padding を 8px に | `padding-top: 8px; padding-bottom: 8px;` |
| `rounded-xl` | 角を大きく丸める | `border-radius: 12px;` |
| `font-bold` | 太字にする | `font-weight: 700;` |
| `flex` | フレックスボックスレイアウト | `display: flex;` |
| `gap-3` | 子要素の間隔を 12px に | `gap: 12px;` |
| `text-xs` | 文字を小さく | `font-size: 12px;` |
| `w-full` | 幅を 100% に | `width: 100%;` |
| `hidden` | 非表示 | `display: none;` |
| `sm:block` | 画面幅 640px 以上で表示 | `@media (min-width: 640px) { display: block; }` |

### 本アプリでの実例

```jsx
// src/tactics-board_1.jsx のレイアウト部分
<div className="flex gap-5 flex-col lg:flex-row">
```

これは以下の意味です：

- `flex` — 子要素を並べるレイアウト
- `gap-5` — 子要素の間隔 20px
- `flex-col` — 縦に並べる（デフォルト）
- `lg:flex-row` — 画面が大きい時は横に並べる

`lg:` は「画面幅 1024px 以上の時」を意味するプレフィックスです。これだけでスマホは縦並び、PC は横並びのレスポンシブデザインが実現できます。

### Tailwind の設定ファイル

```javascript
// tailwind.config.js
export default {
  content: ['./index.html', './src/**/*.{js,jsx}'],  // ← Tailwind がクラス名を探すファイル
  theme: { extend: {} },
  plugins: [],
};
```

`content` で指定したファイルの中から使われているクラス名だけを抽出し、最終的な CSS に含めます。使っていないクラスは含まれないので、ファイルサイズが小さく済みます。

### Tailwind を動かすための裏方：PostCSS

```javascript
// postcss.config.js
export default {
  plugins: {
    tailwindcss: {},     // Tailwind のクラスを CSS に展開
    autoprefixer: {},    // ブラウザ互換のための接頭辞を自動付与
  },
};
```

```css
/* src/index.css — Tailwind のエントリポイント */
@tailwind base;        /* ブラウザ間の差異をリセット */
@tailwind components;  /* コンポーネント用クラス */
@tailwind utilities;   /* px-4, flex 等のユーティリティクラス */
```

この 3 行が、ビルド時に実際の CSS に展開されます。

## 7. lucide-react — アイコンの部品集

### アイコンライブラリとは

UI に使う小さな絵（アイコン）を、自分で画像ファイルを作る代わりに、パッケージとして読み込んで使えるものです。

### 使い方

```jsx
// src/tactics-board_1.jsx 1行目付近
import {
  Shield,          // 盾のアイコン → 守備シーン
  Swords,          // 剣のアイコン → 攻撃シーン
  Save,            // フロッピーディスク → 保存
  Trash2,          // ゴミ箱 → 削除
  Eye, EyeOff,     // 目 → 表示/非表示
  Pencil,          // 鉛筆 → 編集
  Camera,          // カメラ → 写真アップロード
} from 'lucide-react';

// 使う時は HTML タグのように書く
<Shield size={15} />        // 15px の盾アイコンを表示
<Trash2 size={13} />        // 13px のゴミ箱アイコンを表示
```

### なぜ画像ファイルではなくライブラリを使うのか

| 比較 | 画像ファイル（PNG等） | lucide-react |
|------|----------------------|-------------|
| サイズ変更 | 拡大するとぼやける | `size={30}` で綺麗に拡大 |
| 色の変更 | 別の画像を用意する必要がある | CSS で自由に変更可能 |
| ファイル管理 | アイコンの数だけファイルが増える | `import` 1行で済む |
| 読み込み速度 | ファイルごとにリクエスト | JavaScript に含まれる |

内部的には SVG（次のセクションで解説）で描画されているため、どんなサイズでも鮮明です。

## 8. Vite — コードをブラウザ用に変換する道具

### なぜ変換が必要なのか

開発者が書いたコードには、ブラウザがそのまま理解できないものが含まれています。

| 開発者が書くもの | ブラウザが理解できるもの |
|-----------------|----------------------|
| JSX (`<Button />`) | `React.createElement(...)` |
| `import` 文 | ファイルを1つに結合 |
| Tailwind のクラス | 実際の CSS |

Vite はこの変換を行う**ビルドツール**です。

### 開発時と本番時の違い

```bash
# 開発時: ファイルを保存するたびにブラウザが即座に更新される
npm run dev

# 本番用: すべてのファイルを最適化して dist/ フォルダに出力
npm run build
```

**開発時（`npm run dev`）の動き：**

```
1. Vite がローカルサーバーを起動（http://localhost:5173）
2. ブラウザがファイルを要求する
3. Vite がその場で JSX → JS に変換して返す
4. ファイルを保存すると、変更部分だけブラウザに送る（HMR）
5. 画面がページ全体のリロードなしで更新される
```

HMR（Hot Module Replacement）のおかげで、コードを書き換えるとほぼ一瞬で画面に反映されます。

### 設定ファイル

```javascript
// vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],    // ← React（JSX）を変換するプラグイン
});
```

たった 4 行です。Vite は「設定なしでほぼ動く」ことを設計思想としています。

## 9. npm — ライブラリを管理する仕組み

### npm とは

npm（Node Package Manager）は、世界中の開発者が公開しているライブラリ（パッケージ）をダウンロード・管理する仕組みです。Node.js をインストールすると一緒に入ります。

### package.json — アプリのレシピ

```json
{
  "dependencies": {
    "lucide-react": "^0.468.0",
    "react": "^18.3.1",
    "react-dom": "^18.3.1"
  },
  "devDependencies": {
    "@vitejs/plugin-react": "^4.3.4",
    "tailwindcss": "^3.4.15",
    "vite": "^5.4.11"
  }
}
```

| 区分 | 意味 | このアプリの例 |
|------|------|---------------|
| `dependencies` | アプリの動作に必要なもの | React, lucide-react |
| `devDependencies` | 開発時だけ必要なもの | Vite, Tailwind |

### よく使うコマンド

| コマンド | やること |
|----------|---------|
| `npm install` | `package.json` に書かれたライブラリをすべてダウンロードして `node_modules/` に入れる |
| `npm run dev` | `package.json` の `scripts.dev` に書かれたコマンド（`vite`）を実行 |
| `npm run build` | `package.json` の `scripts.build` に書かれたコマンド（`vite build`）を実行 |

### node_modules とは

`npm install` を実行すると作られるフォルダで、ダウンロードしたライブラリの実体が入っています。このフォルダは非常に大きく（数百 MB になることも）、`.gitignore` で Git の管理対象から除外しています。

```
# .gitignore
node_modules    ← Git に含めない
```

別の PC でプロジェクトを動かす時は `npm install` を再実行すれば同じものが入ります。

## 10. localStorage — ブラウザの中のデータ倉庫

### localStorage とは

ブラウザが持っている簡易的なデータ保存場所です。キー（名前）と値（文字列）のペアで保存します。

```javascript
// 保存
localStorage.setItem('username', '太郎');

// 読み込み
const name = localStorage.getItem('username');  // '太郎'

// 削除
localStorage.removeItem('username');
```

### 本アプリでの使われ方

本アプリでは `localStorage` を直接使わず、`storage-polyfill.js` を経由しています。

```javascript
// src/storage-polyfill.js
const PREFIX = 'ftb_storage:';

window.storage = {
  async set(key, value) {
    localStorage.setItem(`${PREFIX}${key}`, value);
    //                    ^^^^^^^^^^^^^^^^
    //                    'ftb_storage:save:1' のように接頭辞を付けて保存
  },
  async get(key) {
    const v = localStorage.getItem(`${PREFIX}${key}`);
    return v != null ? { value: v } : null;
  },
  // ...
};
```

**なぜ間に1枚はさむのか：**

- `ftb_storage:` という接頭辞を付けることで、他のアプリのデータと混ざらない
- `async`（非同期）にしておくことで、将来 IndexedDB やサーバー保存に切り替えやすい
- アプリ本体は `window.storage` だけ知っていればよく、保存先の詳細を知らなくて済む

### localStorage の注意点

| 項目 | 内容 |
|------|------|
| 容量 | 約 5〜10MB（写真を Data URL で保存すると消費が早い） |
| 消える条件 | 「サイトデータを削除」「キャッシュクリア」で消えることがある |
| 共有範囲 | 同じブラウザ・同じ URL（ドメイン + ポート）でのみ読める |
| データ型 | 文字列のみ。オブジェクトは `JSON.stringify()` で文字列に変換して保存 |

## 11. SVG — コードで絵を描く

### SVG とは

SVG（Scalable Vector Graphics）は、コード（テキスト）で図形を描く仕組みです。拡大してもぼやけない**ベクター画像**です。

### 本アプリでのSVG

ピッチの線（外枠・ペナルティエリア・センターサークルなど）はすべて SVG で描かれています。

```jsx
// src/tactics-board_1.jsx のピッチ線描画部分
<svg viewBox="0 0 50 68">
  {/* 外枠 */}
  <rect x={0.3} y={0.3} width={49.4} height={67.4}
        fill="none" stroke="white" strokeWidth={0.3} />

  {/* ハーフウェイライン */}
  <line x1={0.3} y1={34} x2={49.7} y2={34}
        stroke="white" strokeWidth={0.3} />

  {/* センターサークル */}
  <circle cx={25} cy={34} r={6.5}
          fill="none" stroke="white" strokeWidth={0.3} />
</svg>
```

### 基本的な SVG 要素

| 要素 | 描けるもの | 主な属性 |
|------|-----------|----------|
| `<rect>` | 四角形 | x, y, width, height |
| `<circle>` | 円 | cx（中心x）, cy（中心y）, r（半径） |
| `<line>` | 線 | x1, y1（始点）, x2, y2（終点） |
| `<path>` | 自由な形 | d（パスデータ） |

### viewBox の仕組み

```jsx
<svg viewBox="0 0 50 68">
//            x  y  幅  高さ
```

`viewBox` は「この座標系で描きます」という宣言です。実際の表示サイズ（px）がいくつであっても、SVG の中では 幅50 × 高さ68 の世界で描画します。

本アプリでは、シナリオ切替時に `viewBox` を変えることでズームを実現しています。

```javascript
// 試合全体: ピッチ全体を表示
viewBox = "0 0 50 68"

// 守備シーン: 自陣（下半分）だけ表示 → 拡大される
viewBox = "0 34 50 34"

// 攻撃シーン: 敵陣（上半分）だけ表示 → 拡大される
viewBox = "0 0 50 34"
```

画像を切り取っているのではなく、「カメラで映す範囲を変えている」イメージです。

## 12. 技術同士のつながり — 全体マップ

最後に、これらの技術がどう連携しているかを図にまとめます。

```
┌─────────────────────────────────────────────────────────┐
│  開発者が書くファイル                                      │
│                                                          │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │ .jsx ファイル  │  │ index.css    │  │ index.html    │  │
│  │ (React + JSX) │  │ (Tailwind)   │  │               │  │
│  └──────┬───────┘  └──────┬───────┘  └───────────────┘  │
│         │                  │                              │
└─────────┼──────────────────┼──────────────────────────────┘
          │                  │
          ▼                  ▼
┌─────────────────────────────────────────┐
│  Vite（ビルドツール）                     │
│                                          │
│  JSX → JavaScript に変換                 │
│  Tailwind クラス → CSS に変換（PostCSS）  │
│  import を解決して必要なファイルを結合     │
│  HMR で変更をブラウザに即反映             │
└────────────────────┬────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────┐
│  ブラウザ                                │
│                                          │
│  ┌──────────┐  ┌──────────┐             │
│  │ React    │  │ CSS      │             │
│  │ 状態管理  │→│ 見た目    │ → 画面表示  │
│  │ (Hooks)  │  │(Tailwind)│             │
│  └────┬─────┘  └──────────┘             │
│       │                                  │
│       ▼                                  │
│  ┌──────────────────────────────┐        │
│  │ localStorage                 │        │
│  │ (storage-polyfill.js 経由)   │        │
│  │ → 作戦データの保存・読み込み   │        │
│  └──────────────────────────────┘        │
└─────────────────────────────────────────┘
```

### 各技術の担当領域

| 技術 | 何をするか | サッカーで例えると |
|------|-----------|-------------------|
| **JavaScript** | すべての処理の土台 | ピッチそのもの |
| **JSX** | 画面の構造を記述 | フォーメーション図 |
| **React** | 状態に応じて画面を更新 | 監督（指示を出す人） |
| **Hooks** | 状態の管理・副作用の処理 | 監督の戦術ノート |
| **Tailwind CSS** | 見た目を整える | ユニフォームのデザイン |
| **lucide-react** | アイコン表示 | エンブレム・マーク |
| **Vite** | コードをブラウザ用に変換 | 遠征のバス（選手を現地に届ける） |
| **npm** | ライブラリの管理 | クラブの事務局（契約や補強の管理） |
| **localStorage** | データの保存 | 戦術ボードの写真を撮って保存 |
| **SVG** | ピッチの線やアイコンを描画 | ピッチに引かれた白線 |
