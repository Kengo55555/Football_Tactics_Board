## 1. Web の仕組み — ブラウザとファイルの関係

Web ページは、突き詰めると **3種類のファイル** でできています。

```
ブラウザ（Chrome, Safari など）が読み込むもの:

  HTML ファイル  → ページの「内容」（文章、ボタン、画像の場所）
  CSS ファイル   → ページの「見た目」（色、大きさ、配置）
  JavaScript    → ページの「動き」（クリックしたら何が起きるか）
```

ブラウザは、この3つを受け取って画面に表示します。逆に言えば、**ブラウザが理解できるのはこの3つだけ**です。このプロジェクトで使う他の技術（React, Tailwind, JSX など）は、最終的にすべてこの3つに変換されてブラウザに届きます。

## 2. HTML — 文書の骨組み

### 一言で言うと

**HTML（HyperText Markup Language）** は、「ここに見出しがある」「ここにボタンがある」「ここに画像がある」という**ページの構造**を記述する言語です。

### どんな見た目か

```html
<!DOCTYPE html>
<html>
  <head>
    <title>作戦ボード</title>
  </head>
  <body>
    <h1>サッカー作戦ボード</h1>
    <p>選手を動かして作戦を考えよう</p>
    <button>保存</button>
  </body>
</html>
```

`<タグ名>` と `</タグ名>` で内容を挟むのが基本ルールです。

### 主なタグ

| タグ | 意味 |
|------|------|
| `<h1>` | 大見出し |
| `<p>` | 段落（文章のかたまり） |
| `<button>` | ボタン |
| `<img>` | 画像 |
| `<div>` | 汎用の箱（グループ分けに使う） |
| `<a>` | リンク |

### HTML だけだとどうなるか

HTML だけで作ったページは、白背景に黒文字が並ぶだけの素朴な見た目になります。色を付けたり、レイアウトを整えたりするには **CSS** が必要です。

## 3. CSS — 見た目を決めるルール

### 一言で言うと

**CSS（Cascading Style Sheets）** は、HTML で作った骨組みに対して「色」「大きさ」「配置」「フォント」などの**見た目のルール**を指定する言語です。

### どんな見た目か

```css
/* ボタンの見た目を指定する */
button {
  background-color: blue;    /* 背景色を青に */
  color: white;              /* 文字色を白に */
  padding: 10px 20px;        /* 内側の余白 */
  border-radius: 8px;        /* 角を丸く */
  font-size: 16px;           /* 文字の大きさ */
}
```

「**何に対して**（button）」「**何を**（background-color）」「**どうする**（blue）」の3つを書くのが基本です。

### CSS がないと / あると

```
CSS なし:  [ 保存 ]          ← ブラウザ標準の味気ないボタン
CSS あり:  [ 💙 保存 ]       ← 青くて角丸の見やすいボタン
```

### HTML と CSS の関係

HTML が「家の間取り図」だとすると、CSS は「壁紙の色や家具の配置」です。構造（HTML）と見た目（CSS）を分離することで、同じ内容でもデザインだけ変更できます。

## 4. JavaScript — ブラウザに「動き」を与える言語

### 一言で言うと

**JavaScript** は、Web ページに**動的な振る舞い**を追加するためのプログラミング言語です。ブラウザの中で直接動く唯一のプログラミング言語です。

### なぜ必要か

HTML と CSS だけでは「静的な」ページしか作れません。

```
HTML + CSS だけでできること:
  - 文章や画像の表示
  - ボタンの見た目を整える
  - マウスを乗せた時の色変化（簡単なもの）

JavaScript を加えるとできること:
  - ボタンを押したらデータを保存する
  - 選手をドラッグで動かす
  - 入力内容に応じて画面の一部を書き換える
  - タイマーやアニメーション
```

### 基本的な文法

```javascript
// 1. 変数 — データに名前を付けて覚えておく
const teamName = "FC バルセロナ";   // 変わらない値
let score = 0;                      // 後で変わる値

// 2. 関数 — 処理のまとまりに名前を付ける
function addGoal() {
  score = score + 1;
}

// 3. 条件分岐 — 場合によって処理を変える
if (score >= 3) {
  console.log("大量得点!");
}

// 4. 繰り返し — 同じ処理を何度もやる
for (let i = 0; i < 8; i++) {
  console.log(`選手${i + 1}を配置`);
}
```

### 他の言語との比較

| 言語 | 主な用途 | ブラウザで動くか |
|------|---------|-----------------|
| **JavaScript** | Web ページの動作 | **直接動く** |
| Python | データ分析、AI、サーバー | 動かない |
| Java | 業務システム、Android アプリ | 動かない |
| Swift | iPhone アプリ | 動かない |

ブラウザで動くアプリを作る場合、JavaScript は事実上の唯一の選択肢です。

## 5. Node.js — JavaScript をパソコン上で動かす仕組み

### 一言で言うと

**Node.js** は、もともとブラウザの中でしか動かなかった JavaScript を、**パソコン上で直接実行できるようにした実行環境**です。

### なぜブラウザの外で JavaScript を動かすのか

このプロジェクトでは、コードを書いてからブラウザで見るまでに「変換」の工程があります。

```
開発者が書くコード（JSX、Tailwind）
      ↓  ← ここの変換を Node.js が行う
ブラウザが読めるコード（JavaScript、CSS）
```

この変換作業はブラウザではなく、パソコン上で行います。そのために Node.js が必要です。

### Node.js と npm の関係

Node.js をインストールすると、**npm** というツールも一緒に入ります（次のセクションで詳しく説明します）。

```
Node.js をインストール
  ├── node コマンド  → JavaScript を実行する
  └── npm コマンド   → ライブラリを管理する
```

### 重要なポイント

Node.js は**開発中に使うもの**であり、完成したアプリをユーザーが見る時には不要です。ユーザーのブラウザには変換済みのファイルだけが届きます。

## 6. npm — ライブラリの宅配便

### 一言で言うと

**npm（Node Package Manager）** は、世界中の開発者が作って公開している**ライブラリ（便利なプログラムの部品）をダウンロードして管理する仕組み**です。

### ライブラリとは

料理で例えると:

```
全部自分で作る場合:
  小麦粉 → 生地をこねる → 発酵させる → 焼く → パン完成

ライブラリを使う場合:
  スーパーでパンを買う → 完成
```

プログラミングでも、「よく使う機能」を毎回ゼロから書くのは大変です。他の開発者が作って公開してくれている部品（ライブラリ）を使うことで、自分のアプリに必要な部分の開発に集中できます。

### package.json — 「買い物リスト」

プロジェクトに必要なライブラリの一覧は `package.json` というファイルに書かれています。

```json
{
  "dependencies": {
    "react": "^18.3.1",        // 画面の自動更新ライブラリ
    "lucide-react": "^0.468.0" // アイコン集
  },
  "devDependencies": {
    "vite": "^5.4.11",         // 開発ツール
    "tailwindcss": "^3.4.15"   // CSS ツール
  }
}
```

| 区分 | 意味 | 例え |
|------|------|------|
| `dependencies` | アプリの動作に必要 | 料理の材料 |
| `devDependencies` | 開発中だけ必要 | 調理器具 |

### 基本的な使い方

```bash
npm install     # package.json の全ライブラリをダウンロード
npm run dev     # 開発サーバーを起動
npm run build   # 本番用ファイルを出力
```

`npm install` を実行すると `node_modules` というフォルダにライブラリの実体がダウンロードされます。このフォルダは非常に大きいため Git には含めず、必要な時に `npm install` で再取得します。

## 7. React — 画面の「部品化」と「自動更新」

### 一言で言うと

**React** は、Facebook（現 Meta）が開発した **UI（画面）を効率よく作るための JavaScript ライブラリ**です。

### React がなかったら

素の JavaScript で画面を更新するコードを想像してみてください。

```javascript
// 選手の名前が変わった時...
document.getElementById("player1-name").textContent = "太郎";
document.getElementById("player1-badge").textContent = "太郎";
document.getElementById("sidebar-player1").textContent = "太郎";
// ← 名前が表示されている場所を全部手動で見つけて書き換える必要がある
```

画面が複雑になるほど、「どこを更新すればいいか」の管理が爆発的に大変になります。

### React があると

```javascript
// データを更新するだけ
setPlayerName("太郎");
// → React が「この名前を使っている画面」を全部自動で書き換えてくれる
```

**「データを変える」と「画面が変わる」が自動で連動する** — これが React の最大の価値です。

### コンポーネント — 画面の部品

React では画面を「コンポーネント」という再利用可能な部品に分けます。

```
スマートフォンの画面で例えると:

  画面全体（アプリ）
  ├── ヘッダー（上部バー）
  ├── コンテンツ（メイン領域）
  │   ├── カード A
  │   ├── カード B
  │   └── カード C
  └── フッター（下部バー）
```

各コンポーネントは独立した部品なので、「カード」の見た目を変えても「ヘッダー」には影響しません。レゴブロックのように組み合わせて画面を構築します。

### state（状態）— React の記憶力

React のコンポーネントは **state（状態）** というデータを持てます。

```
state の例:
  - 現在のズーム: "試合全体" / "守備" / "攻撃"
  - 選手1の位置: x=25, y=50
  - 保存ダイアログ: 開いている / 閉じている
```

state が変わると、React はその state を使っている画面部分を自動で再描画します。開発者は「state をどう変えるか」だけ考えれば、画面の更新は React が引き受けてくれます。

## 8. JSX — HTML と JavaScript を一緒に書く記法

### 一言で言うと

**JSX（JavaScript XML）** は、JavaScript のコードの中に **HTML のようなタグを直接書ける**記法です。React と一緒に使います。

### なぜ必要か

React でボタンを作るとき、JavaScript だけだとこうなります:

```javascript
// 普通の JavaScript（読みにくい）
React.createElement("button", { className: "blue" }, "保存");
```

JSX を使うとこう書けます:

```jsx
// JSX（HTML のように読みやすい）
<button className="blue">保存</button>
```

どちらも全く同じ意味ですが、JSX の方が直感的に「ボタンを作っている」と分かります。

### ブラウザは JSX を読めない

重要なのは、**JSX はブラウザが直接理解できない**ということです。Vite（後述）が JSX を通常の JavaScript に変換してからブラウザに届けます。

```
開発者が書く: <button>保存</button>        ← JSX
      ↓ Vite が変換
ブラウザが受け取る: React.createElement(...)  ← 普通の JavaScript
```

### HTML との違い

JSX は HTML にそっくりですが、いくつか書き方が異なります。

| HTML での書き方 | JSX での書き方 | 理由 |
|---|---|---|
| `class="blue"` | `className="blue"` | `class` は JavaScript の予約語 |
| `style="color: red"` | `style={{ color: 'red' }}` | JavaScript のオブジェクトで指定 |
| `onclick="..."` | `onClick={...}` | キャメルケースに統一 |

## 9. Tailwind CSS — クラス名を並べるだけでデザインする

### 一言で言うと

**Tailwind CSS** は、あらかじめ用意された**短いクラス名を組み合わせて**スタイルを指定する CSS のフレームワークです。

### 普通の CSS との違い

```css
/* 普通の CSS: 自分でルールを書く */
.save-button {
  padding: 8px 16px;
  background-color: blue;
  color: white;
  border-radius: 8px;
}
```

```html
<button class="save-button">保存</button>
```

Tailwind では CSS ファイルを書かず、HTML（JSX）のクラス名で直接指定します。

```html
<!-- Tailwind: クラス名の組み合わせ -->
<button class="px-4 py-2 bg-blue-600 text-white rounded-lg">保存</button>
```

### よく使うクラス名

| クラス名 | 意味 | CSS で書くと |
|----------|------|-------------|
| `px-4` | 左右の余白 16px | `padding-left: 16px; padding-right: 16px` |
| `py-2` | 上下の余白 8px | `padding-top: 8px; padding-bottom: 8px` |
| `bg-blue-600` | 背景色を青に | `background-color: #2563eb` |
| `text-white` | 文字色を白に | `color: white` |
| `rounded-lg` | 角を丸く | `border-radius: 8px` |
| `flex` | 子要素を横に並べる | `display: flex` |
| `w-full` | 幅を100%に | `width: 100%` |
| `hidden` | 非表示 | `display: none` |

### レスポンシブ対応

画面サイズに応じてデザインを変えるのも簡単です。

```html
<div class="flex-col lg:flex-row">
<!--          ↑ 通常は縦並び  ↑ 画面が大きい時は横並び -->
```

`sm:`, `md:`, `lg:` などの接頭辞を付けるだけで、画面サイズごとのスタイルを指定できます。

| 接頭辞 | 画面幅 | 対応デバイス |
|--------|--------|-------------|
| (なし) | すべて | - |
| `sm:` | 640px 以上 | 大きめのスマホ |
| `md:` | 768px 以上 | タブレット |
| `lg:` | 1024px 以上 | ノートPC |
| `xl:` | 1280px 以上 | デスクトップ |

## 10. Vite — 開発をサクサクにするツール

### 一言で言うと

**Vite（ヴィート）** は、開発者が書いたコード（JSX, Tailwind など）を**ブラウザが読める形に変換**し、**開発中の確認を高速にする**ビルドツールです。

### なぜ変換が必要か

おさらいですが、ブラウザが理解できるのは HTML, CSS, JavaScript の3つだけです。

```
ブラウザが読めないもの         ブラウザが読めるもの
─────────────────       ─────────────────
JSX (<Button />)    →   React.createElement(...)
Tailwind (px-4)     →   padding-left: 16px; ...
import 文           →   ファイルを1つに結合
```

Vite はこの変換を自動で行います。

### 開発モードと本番モード

```bash
npm run dev     # 開発モード: ファイル保存 → 即座にブラウザに反映
npm run build   # 本番モード: すべてのファイルを最適化して出力
```

**開発モード**では、コードを1文字変更して保存するだけで、ブラウザが自動的に更新されます（HMR: Hot Module Replacement）。ページ全体のリロードではなく、変更した部分だけが差し替わるため、作業が非常にスムーズです。

### Vite がなかったら

```
Vite なし:
  コード修正 → 手動でビルド → ブラウザをリロード → 確認（数十秒）

Vite あり:
  コード修正 → 保存 → 自動で画面に反映（1秒以内）
```

## 11. SVG — 拡大してもぼやけない絵

### 一言で言うと

**SVG（Scalable Vector Graphics）** は、コード（テキスト）で図形を描く画像形式です。どれだけ拡大しても**ぼやけません**。

### 普通の画像（PNG/JPEG）との違い

```
PNG / JPEG（ラスター画像）:
  → 小さな点（ピクセル）の集まりで絵を表現
  → 拡大するとギザギザ・ぼやける
  → 写真向き

SVG（ベクター画像）:
  → 「ここからここまで線を引く」という命令で絵を表現
  → 拡大してもくっきり
  → 図形・アイコン・グラフ向き
```

### SVG のコード例

```html
<!-- 青い円を描く -->
<svg width="100" height="100">
  <circle cx="50" cy="50" r="40" fill="blue" />
</svg>
```

| 属性 | 意味 |
|------|------|
| `cx="50"` | 円の中心の x 座標 |
| `cy="50"` | 円の中心の y 座標 |
| `r="40"` | 円の半径 |
| `fill="blue"` | 塗りの色 |

### 基本の図形

| タグ | 描けるもの |
|------|-----------|
| `<rect>` | 四角形 |
| `<circle>` | 円 |
| `<line>` | 直線 |
| `<path>` | 自由な形 |

このプロジェクトでは、サッカーピッチの白線（外枠、センターサークル、ペナルティエリアなど）をすべて SVG で描いています。SVG なら画面サイズが変わってもピッチの線がぼやけません。

## 12. lucide-react — すぐ使えるアイコン集

### 一言で言うと

**lucide-react** は、UI で使う小さな絵（アイコン）を**React コンポーネントとして提供する**ライブラリです。

### アイコンとは

アプリの中で見かける小さなマーク（保存のフロッピーディスク、ゴミ箱、目のマークなど）のことです。これらを毎回自分で絵を描いて画像ファイルにするのは手間がかかります。

### 使い方のイメージ

```jsx
import { Save, Trash2, Eye } from 'lucide-react';

// HTML のタグのように使える
<Save size={20} />      // 保存アイコン（20px）
<Trash2 size={16} />    // ゴミ箱アイコン（16px）
<Eye size={16} />       // 目のアイコン（16px）
```

### 画像ファイルとの違い

| 項目 | 画像ファイル（PNG） | lucide-react |
|------|-------------------|--------------|
| 拡大 | ぼやける | くっきり（SVG ベース） |
| 色変更 | 画像を作り直す | CSS で自由に変更 |
| サイズ変更 | 複数サイズの画像が必要 | `size` の数値を変えるだけ |
| 管理 | ファイルが増える | `import` 1行 |

内部的には SVG で描画されるため、サイズや色を自由にコントロールできます。

## 13. localStorage — ブラウザの中の保存箱

### 一言で言うと

**localStorage** は、ブラウザが内蔵している**簡易的なデータ保存場所**です。ブラウザを閉じてもデータが残ります。

### なぜ必要か

Web ページは通常、ブラウザを閉じると全てのデータが消えます。「作った作戦を次に開いた時にも見たい」場合、どこかにデータを保存する必要があります。

```
保存方法の選択肢:

  サーバーに保存  → サーバーの準備が必要（大がかり）
  ファイルに保存  → ユーザーに保存先を選ばせる手間
  localStorage   → ブラウザが勝手にやってくれる（手軽）
```

### 仕組み

「名前（キー）」と「値」のペアでデータを保存します。

```javascript
// 保存する
localStorage.setItem("player1", "太郎");

// 読み出す
localStorage.getItem("player1");  // → "太郎"

// 削除する
localStorage.removeItem("player1");
```

### 注意点

| 項目 | 内容 |
|------|------|
| 容量 | 約 5〜10MB（写真を大量に保存すると足りなくなる） |
| 共有 | 同じブラウザ・同じサイトでだけ読める |
| 消失リスク | 「サイトデータを削除」すると消える |
| データ型 | 文字列のみ（オブジェクトは変換が必要） |

サーバーに比べると制約はありますが、サーバー不要で手軽にデータを保存できるのが大きな利点です。

## 14. 全体のつながり

これらの技術がどう連携してアプリを動かしているかの全体像です。

```
【開発時（開発者のパソコン上）】

  開発者がコードを書く
    ├── .jsx ファイル ─── React + JSX + JavaScript
    ├── index.css ────── Tailwind CSS の指定
    └── index.html ──── ページの骨組み
                │
        Node.js + npm で管理
                │
                ▼
          Vite が変換
    JSX → JavaScript / Tailwind → CSS
                │
                ▼

【実行時（ユーザーのブラウザ上）】

        ブラウザが受け取る
    ├── JavaScript（React が動く）
    ├── CSS（Tailwind が変換したもの）
    └── HTML
                │
                ▼
        React が画面を組み立て
    ├── SVG でピッチの線を描画
    ├── lucide-react でアイコン表示
    └── localStorage でデータ保存
                │
                ▼
        ユーザーが操作するたびに
        React が必要な部分だけ再描画
```

### 技術を料理に例えると

| 技術 | 料理での役割 |
|------|-------------|
| **HTML** | 料理のレシピ（材料リスト） |
| **CSS** | 盛り付けのルール |
| **JavaScript** | 調理の手順（加熱、混ぜるなどの動作） |
| **React** | 自動調理ロボット（材料が変わったら勝手に作り直す） |
| **JSX** | レシピの簡単な書き方（略語で書ける） |
| **Tailwind CSS** | 盛り付けテンプレート集 |
| **Vite** | 食材を下ごしらえしてくれるアシスタント |
| **Node.js** | キッチン（調理する場所） |
| **npm** | 食材の宅配サービス |
| **SVG** | 型抜き（きれいな形を作る道具） |
| **lucide-react** | デコレーションパーツのセット |
| **localStorage** | 冷蔵庫（作ったものを保存しておく） |
