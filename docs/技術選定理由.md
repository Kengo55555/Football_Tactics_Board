# 技術選定理由 — なぜこの技術を選んだのか

## この資料について

設計書（`docs/設計書.md`）の「2. 技術スタック」に記載された各技術について、**なぜこのアプリに採用したのか**を初心者向けに解説します。

関連ドキュメント:

| ドキュメント | 説明するレベル |
|---|---|
| [技術スタック基礎知識](./技術スタック基礎知識.md) | 「そもそもこの技術って何？」 |
| [技術スタック実践ガイド](./技術スタック実践ガイド.md) | 「このアプリではどう使っている？」 |
| **本書（技術選定理由）** | 「なぜこの技術を選んだ？」 |

技術選定にはいくつかの前提条件がありました。

- **ターゲット**: 親子で使う作戦ボード（子どもがドラッグで選手を動かす等のインタラクティブ操作がある）
- **動作環境**: ブラウザのみ（サーバー不要・インストール不要）
- **開発規模**: 小〜中規模の個人プロジェクト（開発者1〜2名を想定）
- **保守性**: できるだけシンプルに保ち、後から機能追加しやすくする

## 1. JavaScript (JSX)

| 項目 | 内容 |
|------|------|
| 選定理由 | ブラウザでそのまま動く唯一の言語だから |

### なぜ JavaScript なのか

このアプリはブラウザだけで動作します。ブラウザで直接実行できるプログラミング言語は JavaScript だけです。Python や Java など他の言語でも Web アプリは作れますが、それらはサーバー側で動かすものであり、「サーバーなしで HTML を開けばすぐ使える」という要件には JavaScript が最適です。

### JSX とは

JSX は JavaScript の中に HTML のようなタグを書ける拡張構文です。React と組み合わせて使います。

```jsx
// 通常の JavaScript だと
React.createElement('button', { onClick: handleClick }, '保存');

// JSX だと（同じ意味を直感的に書ける）
<button onClick={handleClick}>保存</button>
```

HTML の見た目とロジックを一つのファイルにまとめられるため、「このボタンを押したら何が起きるか」が読みやすくなります。

### TypeScript を使わなかった理由

TypeScript は型チェックにより安全性が上がりますが、小規模プロジェクトでは設定や学習コストに対する恩恵が小さいと判断しました。将来チームが大きくなったり、バグが増えてきた段階で導入を検討すれば十分です。

## 2. React 18（関数コンポーネント + Hooks）

| 項目 | 内容 |
|------|------|
| 選定理由 | インタラクティブな UI を効率よく構築できるから |

### なぜ UI ライブラリが必要なのか

このアプリでは、選手をドラッグしたり、シナリオを切り替えたり、矢印を描いたりと、**画面の状態が頻繁に変わります**。素の JavaScript（Vanilla JS）だけで「状態が変わったら画面のどこを更新するか」を管理するのは、機能が増えるほど複雑になります。

React は「状態（state）が変わったら、必要な部分だけ自動で再描画する」仕組みを提供してくれます。開発者は「今の状態に対してどう見せたいか」だけを書けばよく、更新タイミングの管理を React に任せられます。

### なぜ React なのか（Vue や Svelte ではなく）

| 比較観点 | React | Vue | Svelte |
|----------|-------|-----|--------|
| 利用者数・情報量 | 非常に多い | 多い | 増加中だがまだ少なめ |
| ライブラリの豊富さ | 最も多い | 多い | 少なめ |
| 学習リソース | 日本語含め豊富 | 豊富 | やや少ない |

決定的な差があるわけではありませんが、React は **困ったときに検索で答えが見つかりやすい**という点が、個人開発・初心者にとって大きなメリットです。また lucide-react のようにアイコンライブラリも React 向けが充実しています。

### 「関数コンポーネント + Hooks」とは

React には古い書き方（クラスコンポーネント）と新しい書き方（関数コンポーネント + Hooks）があります。現在の React 公式ドキュメントは関数コンポーネントを推奨しています。

```jsx
// 関数コンポーネントの例
function Counter() {
  const [count, setCount] = useState(0);   // Hooks で状態管理
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

クラス構文より短く書け、`useState` や `useEffect` などの Hooks で必要な機能を組み合わせるスタイルは、初心者にも理解しやすいです。

## 3. Tailwind CSS

| 項目 | 内容 |
|------|------|
| 選定理由 | CSS ファイルを別管理せず、HTML/JSX 上で素早くスタイルを当てられるから |

### 従来の CSS の課題

通常の CSS では、クラス名を考え、別ファイルに定義し、JSX 側で参照する必要があります。

```css
/* styles.css */
.save-button {
  padding: 8px 16px;
  border-radius: 12px;
  font-weight: bold;
}
```

```jsx
<button className="save-button">保存</button>
```

小さなアプリでも CSS ファイルが肥大化しやすく、「このクラス名はどこで使っている？」「名前が被らないか？」といった管理負担が生じます。

### Tailwind のアプローチ

Tailwind は `px-4`（左右padding 16px）、`rounded-xl`（角丸）のような **ユーティリティクラス** をあらかじめ用意しています。

```jsx
<button className="px-4 py-2 rounded-xl font-bold">保存</button>
```

CSS ファイルを書く必要がなく、JSX を見ればスタイルが分かります。このアプリのように **1ファイルで完結** させたい小規模プロジェクトとの相性が良いです。

### インライン style と `.tb-*` クラスの併用

Tailwind だけでは表現しきれないもの（動的な色の計算、クラブカラー定数の参照など）はインライン `style` で補い、hover 時の効果など疑似クラスが必要なものは `<style>` タグで `.tb-*` クラスとして定義しています。単一のツールに固執せず、場面に応じて使い分けている形です。

## 4. lucide-react（アイコン）

| 項目 | 内容 |
|------|------|
| 選定理由 | 軽量で、必要なアイコンだけを読み込めるから |

### アイコンライブラリの役割

UI に「保存」「削除」「シールド（守備）」などのアイコンを表示するために使います。自分で SVG アイコンを作る手間が省けます。

### なぜ lucide-react なのか

| 比較 | lucide-react | Font Awesome | Material Icons |
|------|-------------|--------------|----------------|
| バンドルサイズ | 使ったものだけ | 設定次第で全体が入りがち | やや大きめ |
| React 対応 | ネイティブ | ラッパーが必要 | ラッパーが必要 |
| デザインの統一感 | シンプルで一貫 | 種類が非常に多い | Google 寄り |

lucide-react は **tree-shaking**（使わないアイコンをビルド時に除外する仕組み）が効くため、最終的なファイルサイズが小さく済みます。子どもが使うアプリなのでページの読み込みは速いに越したことはありません。

```jsx
import { Shield, Swords, Save } from 'lucide-react';

// 必要なアイコンだけ import して使う
<Shield size={15} />
```

## 5. Vite + @vitejs/plugin-react（ビルドツール）

| 項目 | 内容 |
|------|------|
| 選定理由 | 設定がほぼ不要で、開発中の表示更新が非常に速いから |

### ビルドツールとは

ブラウザは JSX や Tailwind をそのまま理解できません。ビルドツールは、開発者が書いたコードをブラウザが読める形（普通の JavaScript + CSS）に変換する役割を持ちます。

### なぜ Vite なのか

| 比較 | Vite | webpack | Create React App (CRA) |
|------|------|---------|------------------------|
| 起動速度 | 非常に速い（数百ms） | 遅い（数秒〜数十秒） | 遅め |
| 設定ファイル | ほぼゼロで動く | 複雑になりがち | 隠蔽されている |
| HMR（ホットリロード）| 高速 | 普通 | 普通 |
| 現在の推奨度 | React 公式が推奨 | 成熟しているが古め | 非推奨（2023年〜） |

Vite は `npm run dev` を実行すると数百ミリ秒で開発サーバーが起動し、コードを保存するとブラウザにほぼ即座に反映されます。この**待ち時間の少なさ**は、試行錯誤の多い UI 開発で大きな利点です。

設定ファイル（`vite.config.js`）もこのアプリでは数行で済んでおり、初心者が迷うポイントが少ないです。

## 6. localStorage による永続化（storage-polyfill.js）

| 項目 | 内容 |
|------|------|
| 選定理由 | サーバーやデータベースなしで、ブラウザだけでデータを保存できるから |

### なぜサーバーを使わないのか

サーバーを立てると、ドメイン取得・ホスティング費用・セキュリティ対策・ユーザー認証など、やるべきことが一気に増えます。このアプリは **家族で1台の端末を共有して使う** 想定なので、ブラウザ内にデータを保存する `localStorage` で十分です。

### localStorage の特徴と限界

| 特徴 | 内容 |
|------|------|
| 容量 | 約 5〜10MB（ブラウザにより異なる） |
| 永続性 | ブラウザを閉じても残る（キャッシュ削除で消える） |
| 共有 | 同一ブラウザ・同一ドメインでのみアクセス可能 |
| API | 同期的（`getItem` / `setItem`） |

### なぜ polyfill を挟んでいるのか

アプリ本体は `window.storage` という**非同期の抽象 API**（`list` / `get` / `set` / `delete`）を使う設計です。`storage-polyfill.js` がこれを `localStorage` に変換しています。

```
アプリ → window.storage.set("save:1", data)
                ↓ polyfill が変換
localStorage.setItem("ftb_storage:save:1", data)
```

こうしておくと、将来「Electron アプリにしてファイルに保存したい」「IndexedDB に切り替えたい」となったとき、**アプリ本体のコードを変えずに polyfill 部分だけ差し替えれば済みます**。初期段階からこの分離をしておくことで、後の拡張が楽になります。

## まとめ — 技術選定の判断軸

このプロジェクトの技術選定に共通する判断軸をまとめます。

| 判断軸 | 説明 |
|--------|------|
| **シンプルさ** | 設定や学習コストが少ないものを優先した（Vite, Tailwind） |
| **ブラウザ完結** | サーバー不要で動くことを最優先にした（localStorage, 静的ビルド） |
| **情報の見つけやすさ** | 困ったとき検索で解決策に辿り着ける技術を選んだ（React, Tailwind） |
| **拡張性の確保** | 現時点の複雑さは最小限にしつつ、将来の変更に備えた設計にした（storage polyfill） |
| **ユーザー体験** | 読み込みの速さ・操作の応答性を重視した（Vite の HMR, lucide の tree-shaking） |

技術選定に「唯一の正解」はありません。プロジェクトの規模・チーム・目的に応じて最適な組み合わせは変わります。大切なのは **「なぜその技術を選んだのか」を説明できること** です。
